# CONTRIBUTING_AI.md

Этот документ обязателен для всех ИИ-агентов, вносящих изменения в проект.
Следуй правилам буквально. Не интерпретируй их свободно.

## 1. Source of Truth (Источники знаний)

1. ПЕРЕД ЛЮБЫМ ИЗМЕНЕНИЕМ КОДА ОБЯЗАТЕЛЬНО ПРОЧИТАЙ:
- `README.md`
- `docs/INTERNAL.md`
- `docs/api/INDEX.md`
- профильный доменный файл в `docs/api/*.md` (выбери по роутеру/бизнес-области).

2. ПРИ ИЗМЕНЕНИИ ЛОГИКИ БД ОБЯЗАТЕЛЬНО СВЕРЯЙСЯ С `DATAMODELS.md`.
- Если `DATAMODELS.md` отсутствует или устарел, СНАЧАЛА создай/обнови его, ПОТОМ меняй код.

3. НЕ НАЧИНАЙ ИМПЛЕМЕНТАЦИЮ, пока не зафиксировал, какой файл документации является источником правды для текущей задачи.

## 2. Workflow: "Сначала доки, потом код"

1. Если меняется сигнатура метода, логика эндпоинта, схема request/response, контракт ошибок, auth/authz:
- СНАЧАЛА обнови соответствующий `.md` файл документации.
- ТОЛЬКО ПОСЛЕ этого меняй код.

2. При добавлении нового эндпоинта:
- Немедленно добавь его описание в нужный домен `docs/api/<domain>.md`.
- Обязательно опиши: `Data Contract`, `Authentication & Authorization`, `Logic Flow`, `Error Handling`, `Usage Examples`.

3. При добавлении нового API-домена:
- Создай новый файл в `docs/api/`.
- Добавь домен в `docs/api/INDEX.md` (карта + таблица ссылок).

4. После изменения API-документации:
- Проверь, что навигация в `docs/api/INDEX.md` актуальна.
- Проверь, что ссылки на модели и секции не битые.

## 3. Архитектурные ограничения

1. ДЕРЖИ роутеры тонкими.
- Роутер: валидация входа/выхода, вызов сервиса, трансляция ошибок.
- Бизнес-логика: только в `backend/app/services/*`.

2. ИСПОЛЬЗУЙ Pydantic-схемы для всех API-контрактов.
- Не вводи неформализованные `dict`-контракты без документированного обоснования.

3. ИСПОЛЬЗУЙ строгую типизацию (Type Hinting).
- Добавляй типы для параметров и возвращаемых значений.
- Не оставляй публичные функции без типов.

4. НЕ ЛОМАЙ действующий API-контракт молча.
- Любое breaking change должно быть заранее отражено в документации.

5. ОБРАБАТЫВАЙ ошибки предсказуемо.
- Используй согласованные HTTP-коды.
- Поддерживай единый формат error response (`detail` + validation envelope для `422`).

## 4. Регламент обновлений

1. ПОСЛЕ КАЖДОЙ ФИЧИ проверь необходимость обновления:
- `docs/OPERATIONS.md` (эксплуатация, runbook, инциденты, релизы, фоновые процессы).
- `TECH_DEBT.md` (технический долг, компромиссы, риски).

2. ЛЮБОЙ "КОСТЫЛЬ" фиксируй в `TECH_DEBT.md` НЕМЕДЛЕННО (в той же задаче).
- Укажи: контекст, риск, почему решение временное, условие удаления.

3. Если изменились env-переменные, команды запуска, воркеры, расписания:
- Обнови `README.md` и/или `docs/OPERATIONS.md` в той же задаче.

## 5. Self-Check (обязательный перед завершением)

Перед закрытием задачи задай себе вопрос:

`"Соответствует ли мой код написанной документации и не нарушает ли он API-контракт?"`

Если ответ не "Да" для любого пункта ниже, ЗАДАЧУ НЕ ЗАВЕРШАЙ:

1. Документация обновлена раньше или одновременно с кодом.
2. Измененные эндпоинты отражены в правильном доменном файле `docs/api/*.md`.
3. Контракты request/response и коды ошибок совпадают с реализацией.
4. Архитектурные ограничения (тонкие роутеры, сервисная бизнес-логика, Pydantic, type hints) соблюдены.
5. Обновлены `TECH_DEBT.md` и/или `docs/OPERATIONS.md`, если это требовалось.

## 6. Режим исполнения

1. При конфликте между кодом и документацией:
- Сначала обнови документацию до фактического состояния.
- Затем приведи код к согласованному контракту.

2. При неясности требований:
- Не импровизируй.
- Сформулируй допущения явно и зафиксируй их в документации.

3. Любую задачу считай незавершенной, пока код, документация и API-контракт не синхронизированы.

## 7. Russian Language Support (обязательно)

1. УЧИТЫВАЙ, что проект активно использует русский язык в документации и комментариях.

2. ЗАПРЕЩЕНО при редактировании файлов:
- заменять существующий русский текст на транслит;
- заменять существующий русский текст на Unicode-escape последовательности (например, `\u04xx`).

3. СОХРАНЯЙ русский текст в обычном читаемом виде (UTF-8), без искусственных преобразований.

4. ПЕРЕД ЗАВЕРШЕНИЕМ задачи дополнительно проверь измененные файлы на появление транслита вместо русского и на появление `\u04xx`-escape.
