# Project Prompt Pack For Codex

Документ содержит не внутренние system/developer prompt'ы Codex, а реконструированные проектные prompt-правила, выведенные из бизнес-требований и рабочих указаний по этому проекту.

Использование:
- передавать второму Codex как стартовый контекст;
- использовать как prompt pack для быстрого вхождения в проект;
- дополнять по мере появления новых стабильных правил.

## 1. Главный рабочий prompt

Ты работаешь над CRM/ERP-системой для управления строительными проектами.

Ключевая бизнес-сущность системы — `Сделка / Проект`.

Почти любые новые функции, документы, активности, платежи, задачи, стадии исполнения, панели подрядчиков и заказчиков должны мыслиться как связанные со `Сделкой`, а не как отдельные изолированные модули.

При проектировании и доработке:
- исходи из того, что система живет вокруг проектной карточки;
- не дроби логику по модулям без необходимости;
- проверяй, как изменение влияет на сделки, этапы, договоры, документы, платежи, активность и права доступа;
- если можно выбрать между “отдельным новым блоком” и “проекцией существующих данных сделки”, приоритет у проекции существующих данных.

## 2. Бизнес-модель, которую нужно держать в голове

Система — это не просто CRM, а монолитная ERP/CRM для проектно-строительной компании.

Упрощенная цепочка:

```text
Лид -> Сделка/Проект -> Товары -> Этапы -> Контрактация -> Исполнение
     -> Договоры -> Исходящие / Документы / Файлы
     -> Платежи / Казначейство / ДДС
     -> Активность / Контроль данных / Внешние панели
```

Ключевые стабильные сущности:
- `Сделка`
- `Контрагент`
- `Товар сделки`
- `Этап`
- `Контрактация / назначение`
- `Исполнение`
- `Договор`
- `Исходящий документ`
- `Платеж казначейства`
- `Запись ДДС`
- `Пользователь / роль / суброль / linked_company`

## 3. Prompt по этапам и производственной логике

Думай об исполнении не просто как о наборе этапов.

Правильная логика:
- основное производственное ядро — это связка `этап -> товар -> исполнитель -> назначение`;
- зависимости между этапами должны учитывать не только плановую дату окончания, но и фактическую `дату закрытия`;
- если у этапа есть `close_date`, то рекурсивные зависимости должны ориентироваться в первую очередь на нее;
- если `close_date` отсутствует, используется плановая дата окончания;
- при закрытии этапа пользователь должен иметь возможность задать дату закрытия вручную;
- по умолчанию в форме закрытия должна подставляться сегодняшняя дата;
- если дата закрытия выходит за пределы периода этапа, пользователь должен получать визуальное предупреждение;
- старые закрытые этапы допустимо backfill'ить: `close_date = planned_end_date`, если отдельная дата закрытия ранее не хранилась.

Дополнительно:
- `Gantt` — это визуальная проекция, не источник истины;
- `Календарь` должен показывать прежде всего ключевые/управленческие этапы, а не весь технический шум.

## 4. Prompt по подрядчикам, субролям и панелям

Подрядные сценарии не должны строиться только на обычных ролях.

Обязательная логика:
- у подрядных пользователей доступ определяется через связку `пользователь -> суброль -> контрагент`;
- `linked_company` / привязка к контрагенту — критична;
- исполнитель подрядчика должен видеть только свои товары, этапы, документы и задачи;
- руководитель подрядчика должен видеть расширенный срез того же контура;
- `Панель исполнителя` и `Панель заказчика` — это не отдельные продукты, а ограниченные проекции данных основной проектной модели.

## 5. Prompt по спискам сделок и лидов

Основной рабочий режим по сделкам — списочный.

Правила:
- `Kanban` — вторичен;
- при 3–7 активных объектах список важнее канбана;
- стоит поддерживать пользовательские фильтры и индивидуальную настройку видимых колонок;
- расширение списка новыми полями допустимо, если это действительно повышает обзорность, а не превращает UI в перегруженную таблицу.

Для `Лидов`:
- блок формирования КП вероятно должен уйти в отдельную вкладку;
- у КП нужен предпросмотр или близкий к нему сценарий.

## 6. Prompt по исходящим документам

Исходящие — это не только письма.

Единый реестр должен включать:
- письма;
- счета;
- УПД;
- акты;
- счет-фактуры.

При этом:
- логика у них единая как у исходящего документа;
- отличаются структура полей, шаблоны, нумерация и набор связей.

### 6.1 Нумерация

Для `счетов`, `УПД`, `счет-фактур`:
- номер формируется по каждой нашей компании отдельно;
- формат: `ДДММГГ/00001`;
- счетчик отдельный для каждого типа документа и для каждой компании на текущий день.

Для `актов`:
- номер должен определяться внутри договора;
- следующий акт получает следующий порядковый номер в рамках договора.

### 6.2 Поля и связи

Исходящий документ должен уметь связываться с:
- `Сделкой`;
- `Договором`;
- `Контрагентом`;
- `Нашей компанией`;
- `Этапами`;
- `Платежами`;
- `Шаблоном`;
- `Версиями файла`;
- `Файлами`.

Специфические правила:
- для `акта` нужен выбор этапов и платежей;
- для `УПД` и `счет-фактуры` нужен выбор этапов;
- для `счета` нужен выбор платежа;
- частичные зачеты платежей должны поддерживаться на уровне отдельных сумм;
- один платеж может закрываться разными частями в нескольких документах.

### 6.3 UI-правила по типам исходящих

Для всех типов, кроме писем:
- убрать `Обращение`;
- убрать `Форму обращения`;
- убрать блок файлов приложений как обязательную часть карточки;
- поле `Кому` заменить на `Кем`;
- падеж должен быть родительный.

Для писем:
- оставить существующую логику с дательным падежом;
- банковские реквизиты получателя в письмах не показывать.

### 6.4 Черновики

Черновики исходящих разных типов не должны смешиваться:
- письмо не должно восстанавливаться в счет;
- счет не должен восстанавливаться в акт;
- состояние draft должно храниться с учетом типа документа.

### 6.5 Отражение в сделке

Закрывающие документы должны быть видны обратно в сделке:
- в этапах;
- в связанных сущностях;
- как перекрестные чипы/крошки;
- с типом и номером документа;
- с суммой, если это важно для пользователя.

## 7. Prompt по шаблонам документов

Шаблоны — это не просто загруженные `docx`.

Развитие шаблонов должно идти в сторону:
- админ-вкладки `Шаблоны документов`;
- справочника полей;
- отображения доступных плейсхолдеров;
- online-конструктора шаблонов;
- общей логики с `Outgoing Registry`.

Справочник полей нужен не как full-text поиск по содержимому docx, а как каталог доступных идентификаторов:
- какие поля можно вставлять;
- как они называются;
- из какой сущности берутся.

## 8. Prompt по документному контуру сделки

Во вкладке документации сделки:
- тип документа (`ТЗ`, `ИД`, `Результаты` и т.д.) лучше переключать хлебными крошками/чипами сверху;
- центр должен быть одним explorer-представлением;
- слева — дерево папок;
- справа — табличка файлов.

При большой глубине:
- дерево должно быть видно полностью;
- пользователь должен сам сворачивать/разворачивать ветки;
- клик по папке должен выбирать ее содержимое справа, но не “проваливать” пользователя в отдельный изолированный каталог.

## 9. Prompt по задачам

Задачи — это не только task tracker.

Они должны оставаться универсальным координационным слоем.

Правила:
- поддерживать множественных ответственных;
- поддерживать наблюдателей;
- chat/task activity должна оставаться логически связанной с задачей;
- бюджетирование можно скрыть из UI, но пока оставить на backend;
- аукционы и правила можно временно убрать из UI, но backend лучше не вычищать, пока не принято окончательное решение.

Дополнительно:
- матрица Эйзенхаузера — допустимый альтернативный вид задач;
- статусные и визуальные режимы могут быть индивидуальными по пользователю.

## 10. Prompt по казначейству и ДДС

`Казначейство` и `ДДС` — один из самых устойчивых контуров. Его лучше не ломать без веской причины.

Развитие должно идти так:
- сохраняем базовую модель;
- усложняем работу по компаниям и направлениям;
- допускаем, что не во всех направлениях будет использоваться казначейство целиком;
- ДДС при этом может использоваться отдельно.

UI-правила:
- везде, где есть категория платежа, нужен умный поисковый селектор;
- массовое редактирование по сделке, договору, категории — важный сценарий;
- распределенные платежи не должны визуально шуметь там, где пользователю нужны только рабочие остатки.

## 11. Prompt по контрагентам

Блок `Контрагенты` остается, но UI-карточка должна эволюционировать.

Приоритеты:
- реквизиты;
- документы;
- привязки к нашим компаниям;
- более удобное внутреннее представление связей;
- в карточке компании нужны собственные документы с выбором “по какой нашей компании добавлен файл”.

## 12. Prompt по контролю данных

`Контроль данных` — это отдельный важный слой.

Его нужно развивать как:
- отдельный блок/вкладку;
- системную диагностику;
- перечень проблем по срокам, незакрытым хвостам, товарам без исполнителей, письмам без привязки, договорам без связи и т.д.

Логика:
- база блока остается;
- корректируется состав отслеживаемых проблем;
- важно вести пользователя к первичной сущности, вызвавшей проблему.

## 13. Prompt по активности

`Активность` должна постепенно стать похожей на сквозную ленту проекта.

Требование:
- письма, документы, изменения этапов, платежи, задачи, действия пользователей должны в будущем стекаться в единый activity feed;
- переход из activity должен открывать именно ту сущность, которая вызвала событие, а не “последнюю похожую запись”.

## 16. Prompt по UI/UX

Общие принципы:
- делать интерфейсы компактнее, а не детальнее, если речь идет о рабочих списках;
- не раздувать overview там, где нужен общий взгляд;
- если дерево большое, не заставлять пользователя проваливаться в отдельные уровни без необходимости;
- если пользователь просит “как в битриксе”, ориентироваться на сквозную ленту, фильтры, чипы, компактные таблицы и пользовательские представления;
- изменения UI вводить аккуратно и итеративно: пользователь плохо принимает агрессивные полные редизайны.

## 17. Prompt по технике работы

Рабочие правила по проекту:
- при работе с тестовым контуром не трогать продовую БД;
- если задача касается тестового VPS, changes сначала валидировать на тестовом контуре;
- после выкладки фронта на тестовый сервер проверять доступность `assets`;
- после ручной выкладки править владельца и права, если nginx отдает `403/404`.

## 18. Copy-paste prompt для второго Codex

Ниже короткая версия, которую можно вставить второму Codex как стартовый prompt:

```text
Ты работаешь над ERP/CRM для управления строительными проектами.
Центральная сущность — Сделка / Проект. Почти любые новые функции и данные должны мыслиться как связанные со сделкой.

Ключевые сущности: Контрагент, Товар сделки, Этап, Контрактация/назначение, Исполнение, Договор, Исходящий документ, Платеж, ДДС, Пользователь/Роль/Суброль.

Исполнение нужно мыслить через связку этап -> товар -> исполнитель -> назначение, а не просто через этапы.
Если у этапа есть close_date, зависимости должны ориентироваться на нее; если нет — на плановый end_date.

Исходящие — это единый реестр писем, счетов, УПД, актов и счет-фактур. Для счетов/УПД/счет-фактур номер должен быть в формате ДДММГГ/00001 по типу документа и по нашей компании на текущий день. Акт получает следующий номер внутри договора.

Для всех типов исходящих кроме писем: убрать обращение, форму обращения, банковские реквизиты получателя в письмах не показывать, вместо “Кому” использовать “Кем” в родительном падеже.

Панель исполнителя и панель заказчика — не отдельные продукты, а ограниченные проекции основной модели. Подрядный доступ строится через связку пользователь -> суброль -> контрагент.

Казначейство и ДДС — устойчивое ядро. Их лучше расширять, а не ломать. Везде, где есть категория платежа, нужен умный поисковый селектор.

Контроль данных — отдельный системный блок диагностики. Активность — единая сквозная лента проекта. AI — помощник поверх проектных данных, а не просто чат.

При изменениях проверяй, как они влияют на сделки, этапы, договоры, исходящие, платежи, активность, панели и права доступа.
Не трогай продовую БД при работе с тестовым контуром.
```
