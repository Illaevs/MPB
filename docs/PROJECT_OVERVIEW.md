# Project Overview

Документ дает целостное представление о системе: что это за продукт, какие бизнес-процессы он покрывает, из каких модулей состоит и как устроен на верхнем уровне. Для деталей по API, внутренней реализации и эксплуатации см. `README.md`, `docs/README.md` и `docs/INTERNAL.md`.

## 1. Назначение системы

Система представляет собой монолитную CRM/ERP-платформу для строительных и проектных компаний. Она объединяет в одном контуре:

- продажи и лидогенерацию;
- ведение сделок и проектов;
- планирование этапов и графиков;
- контрактацию и исполнение работ;
- документооборот и исходящую корреспонденцию;
- казначейство, ДДС и экономику;
- взаимодействие с подрядчиками, исполнителями и заказчиками;
- тендерные и аккредитационные процессы;
- внутренние коммуникации, почту и уведомления.

Система рассчитана на работу нескольких ролей одновременно: руководителей, ГИПов, финансового блока, юристов, подрядчиков, исполнителей и заказчиков.

## 2. Что пользователь видит в системе

На уровне интерфейса платформа строится вокруг нескольких ключевых разделов:

- `Leads` и `Projects` для входящего потока и активных сделок;
- `Project Detail` как центральная карточка проекта со стадиями, товарами, договорами, исполнением и графиками;
- `Contracts` и `Contract Detail` для договоров и документов по ним;
- `Tasks` для внутренних задач (с подзадачами, чек-листами, чатом, аукционами);
- `Treasury`, `Income/Expense`, `Finance` для финансового контура;
- `Outgoing Registry`, `Document Registry`, `Files Catalog` для документов и файлового хранения;
- `Executor Panel` для подрядчиков и исполнителей;
- `Customer Panel` как упрощенный кабинет заказчика по его проектам;
- `Home / Feed` — корпоративная лента (посты, опросы, эмодзи-реакции, @mentions, картинки и файлы), плюс дашборд KPI и виджеты;
- `Mail`, `Messenger`, `Notifications` для коммуникаций;
- `Support` (Тех. поддержка) — тикеты пользователей и рабочее место сотрудника поддержки;
- `Reglaments` — нормативная база (СП / ГОСТ / СНиП) с FTS5 + семантическим поиском;
- `My Profile`, `Workday`, `Absences` — HR-блок: расширенный профиль сотрудника, рабочее время с накопительным счётчиком за день, единая таблица отсутствий с таймлайном;
- `Search` (`/search` + Ctrl+K «командная палитра») — глобальный гибридный поиск по всем сущностям с ACL-фильтром;
- `Approvals` / `My Approvals` — согласования по проектам и задачам;
- `Integrations Outbox` / `Subscriptions` — наблюдаемость Event Bus v2 (исходящие события и подписки).

## 3. Верхнеуровневая архитектура

### 3.1 Frontend

Frontend реализован как SPA на:

- `Vue 3`
- `Vue Router`
- `Pinia`
- `Axios`
- `Vite 6`

Клиентский код живет в `frontend/src` и делится на:

- `views/` — экранные модули;
- `components/` — переиспользуемые UI-компоненты;
- `stores/` — глобальные состояния, сессия, очереди;
- `services/` — HTTP и инфраструктурные сервисы;
- `utils/` и `composables/` — прикладные утилиты и composable-логика.

Интерфейс построен вокруг одного каркаса приложения (`App.vue`) с боковой навигацией, глобальными уведомлениями, глобальным чатом, очередями загрузок и общими модальными инструментами.

### 3.2 Backend

Backend реализован на:

- `FastAPI`
- `SQLAlchemy 2`
- `Pydantic 2`
- `PyJWT`
- `Redis` для части security/rate-limit сценариев

Код backend расположен в `backend/app` и разделен на:

- `routers/` — REST endpoints;
- `models/` — ORM-модели;
- `schemas/` — контракты API;
- `services/` — доменная и инфраструктурная логика;
- `core/` — конфигурация, безопасность, middleware;
- `database/` — сессии и база.

Основной API публикуется под `/api/v1`.

### 3.3 Хранилище и фоновые процессы

Система использует файловое хранилище для:

- документов по договорам;
- исходящих писем и их версий;
- файлов исполнителей;
- результатов работ;
- вложений задач и переписки;
- общего файлового каталога.

Фоновые процессы:

- `upload_worker.py` — обработка очереди загрузок;
- `notifications_worker.py` — уведомления, overdue-сценарии, digest;
- `mail_worker.py` — синхронизация почтовых ящиков;
- `event_outbox_worker.py` — Event Bus v2: транзакционный outbox, retry + DLQ, JSON-Logic фильтрация, HMAC-подпись для подписчиков;
- `embedding_worker.py` — гибридный поиск: вычисление bge-m3 embedding'ов (1024-dim) для FTS5-индексированных сущностей.

## 4. Модель доступа и ролей

Доступ в систему строится по section-based матрице прав. Для каждого раздела роль может иметь разные уровни чтения/доступа, например:

- `read_all`
- `read_assigned`
- отдельные write-операции через backend guards

На практике система поддерживает набор рабочих ролей и типов привязки:

- суперпользователь и администратор;
- ГИП и внутренние пользователи;
- руководитель подрядчика и сотрудник подрядчика;
- заказчик;
- предметные роли по конкретным разделам: финансы, договоры, юристы и т.д.

Дополнительно используются связи пользователя с компаниями:

- как руководителя;
- как сотрудника;
- как заказчика.

Это особенно важно для `Executor Panel` и `Customer Panel`, где набор доступных объектов строится не по ручному выбору компании, а по фактическим привязкам пользователя.

## 6. Основные бизнес-модули

### 6.1 Leads и Projects

`Leads` используются как входной контур предпродажной работы. После перевода лид становится `Project/Deal`.

Карточка сделки — центральный узел системы. В ней сосредоточены:

- общие реквизиты проекта;
- товары сделки;
- этапы;
- договоры;
- платежи;
- контрактация;
- исполнение;
- gantt-графики;
- связанные подрядчики и привязки.

### 6.2 Этапы, план-график и исполнение

В системе есть два разных, но связанных слоя:

- плановая структура проекта;
- фактическое исполнение по товарам и исполнителям.

Для проекта используются:

- этапы и зависимости между ними;
- gantt по этапам;
- gantt по контрактации;
- de-facto представление `этап -> товар -> assignment -> подзадачи`.

Для assignment поддерживаются отдельные сроки:

- дата начала;
- рабочий срок;
- договорной срок.

Это позволяет строить разные срезы прогресса:

- для внутреннего управления;
- для исполнителей;
- для заказчика.

### 6.3 Contracts и контрактация

Модуль договоров покрывает:

- реестр договоров;
- карточку договора;
- документы внутри договора;
- статус договора как сущности;
- статусы документов договора;
- привязку договора к проекту;
- финансовую сводку и выплаты.

Внутри проекта вкладка договоров и контрактация связывают договорную структуру с этапами, товарами и исполнением.

### 6.4 Tasks и аукционы

Модуль задач — это не только простой task-tracker. Он используется для:

- внутренних задач;
- задач по проектам;
- задач по исполнению;
- аукционов между подрядчиками;
- обмена файлами и комментариями;
- чата по задаче.

### 6.5 Treasury, ДДС, Finance

Финансовый контур состоит из нескольких взаимосвязанных частей:

- `Treasury` — импорт банковских выписок, правила авторазбора, распределение платежей;
- `Income/Expense` — записи ДДС;
- `Finance` — финансовый учет и агрегированные показатели.

Система умеет:

- импортировать выписки;
- автоматически применять treasury-правила;
- создавать/связывать записи ДДС;
- связывать финансовые движения с проектами, договорами и работами.

### 6.6 Документы и исходящая корреспонденция

Документный контур включает:

- `Outgoing Registry` — исходящие письма, версии, PDF/DOCX, отправка;
- `Document Registry` — документы по каналам и пакетам;
- `Files Catalog` — файловое хранилище как отдельный рабочий раздел;
- storage-модуль для навигации по файловой структуре.

Это один из самых насыщенных по сценариям блоков системы: документы здесь не просто загружаются, а участвуют в согласованиях, версиях, выгрузках, пакетах и заказных маршрутах.

### 6.7 Mail, Messenger и уведомления

В системе есть сразу несколько коммуникационных каналов:

- почта (`Mail`) с OAuth/IMAP/SMTP сценариями;
- глобальный мессенджер;
- task chat;
- центр уведомлений;
- глобальные всплывающие уведомления.

Уведомления генерируются как от действий пользователя, так и от фоновой логики по событиям.

### 6.8 Tenders, аккредитации, юрблок

Дополнительные, но важные бизнес-блоки:

- тендеры и предложения подрядчиков;
- аккредитации подрядчиков и их документы;
- `Legal Work` для юридических кейсов;
- review/approval-потоки по результатам работ.

### 6.9 Executor Panel

Панель исполнителя — это специализированный интерфейс для подрядчиков и исполнителей. Она показывает:

- только связанные объекты;
- этапы и товары конкретного исполнителя;
- подзадачи;
- документы, результаты и папки хранения;
- gantt по рабочим срокам.

Это не просто read-only панель, а рабочее место исполнителя.

### 6.10 Customer Panel

Кабинет заказчика — отдельный упрощенный интерфейс по проектам заказчика. Он показывает:

- объект и адрес;
- сводку по проекту;
- документы;
- письма только от нашей компании к заказчику;
- платежную сводку;
- gantt по товарам и статусам.

Этот раздел специально спроектирован как внешний, более компактный и понятный слой поверх данных проекта.

### 6.11 Тех. поддержка (тикет-система)

Раздел `Support` — внутренняя тикет-система. Он покрывает два сценария на одном наборе данных:

- заявитель: создаёт тикет (тема, описание, категория, файлы), ведёт переписку, видит только свои тикеты и только публичные сообщения;
- сотрудник поддержки (право `read_all` по секции `support` или суперюзер): видит все тикеты, внутренние заметки, меняет статус/категорию/исполнителя и может создать задачу из тикета.

Категории тикетов: ошибка, улучшение, доступ, вопрос, прочее. Чат тикета переиспользует механику чата задач (вложения, вставка из буфера). Весь модуль гейтится одной секцией прав `support`, а разграничение «админ/пользователь» внутри неё — по флагу `read_all`.

### 6.12 Персонализация интерфейса

Пользовательские настройки интерфейса хранятся как единый гибрид: мгновенный кэш в `localStorage` плюс синхронизация с бэкендом (`user.ui_preferences`, endpoints `GET`/`PATCH /api/v1/users/me/ui-preferences`). Это даёт перенос настроек между браузерами и устройствами. Покрывается: тема и акцент, плотность и масштаб, обои, стартовая страница, частота опроса уведомлений, закрепление и порядок пунктов навигации, конфигурация дашборда и сохранённые представления таблиц.

### 6.13 Корпоративная лента (Feed)

`Home` — основная страница сразу после входа: лента новостей компании, KPI-блок, виджеты «Мои задачи», «Активные проекты», «Дни рождения», «Сегодня отсутствуют», популярные посты.

В ленте:

- посты (новости / обычные записи), опросы (single/multi, аноним/открытый), картинки и произвольные файлы (PDF/DOCX/XLSX/архивы/CAD), `@mentions` с автокомплитом по сотрудникам;
- опросы поддерживают дедлайн (`poll.closes_at`, хранится UTC-ISO) и досрочное закрытие (`POST /feed/{id}/poll/close`, право — автор или `feed.edit_all`); после дедлайна/закрытия голосование блокируется на сервере (403), показываются итоги (победитель/проценты, `poll.result`);
- эмодзи-реакции на пост (Slack-стиль, несколько разных эмодзи от одного пользователя), комментарии, счётчик просмотров; автор поста получает in-app уведомление о реакции (тип `feed_reaction`, с антиспамом — пока предыдущее не прочитано, новое не плодится);
- вкладки-фильтры ленты: Все / Новости / Опросы / Упоминания (`GET /feed?tab=...`); вкладка «Упоминания» работает по обратному индексу `feed_mentions` (заполняется при создании/правке поста и в комментариях, покрывает и упоминания в комментариях);
- лайв-плашка «N новых записей» — лёгкий polling `GET /feed/since?after=<created_at>` (раз в 30 c, пауза при скрытой вкладке); новые посты вливаются в ленту по клику, чтобы не сбивать позицию чтения;
- inline-редактор поста (текст + упоминания + добавление/удаление attachments), индикатор «изменено», закрепление наверх;
- права: публикация/правка/удаление — только роль с `feed.edit_all`; чтение/комментарии/реакции/голосование — любой авторизованный.

Attachments хранятся раздельно: картинки в `static/feed/`, файлы — в `static/feed-files/`. `FeedAttachment.kind` (`"image"` / `"file"`) и `size` указаны в схеме; у старых attachments `kind` отсутствует и считается `"image"` по умолчанию.

Миграция новой таблицы упоминаний — `backend/migrate_add_feed_mentions.py` (идемпотентна; разовый backfill существующих упоминаний выполняется только при первом создании таблицы). Поля опроса (`closes_at`/`closed`/`closed_at`/`result`) живут в JSON-колонке `FeedPost.poll` — ALTER не требуется.

### 6.14 Нормативная база (Reglaments)

Каталог СП / ГОСТ / СНиП с двумя пользовательскими сценариями: поиск по реквизитам и тематический поиск.

- модель `Reglament` (документ) + `ReglamentSection` (разделы для cover-section pattern у норм без разделов);
- изолированный поисковый домен: `reglament_fts` + `reglament_embeddings` (bge-m3), не пересекается с основной CRM-выдачей;
- бутстрап парсером (`reglament_parser.py`) с PyMuPDF в приоритете над pypdf (в ~10 раз лучшее качество извлечения для отсканированных PDF);
- batch-скрапер с YAML-конфигом (`reglaments_data/*.yaml`) для пакетной загрузки;
- админский UI загрузки новой нормы (Phase 3);
- если для источника не нашли проверенный URL — кнопка «Открыть в Google» вместо вранья о ссылке.

### 6.15 HR-блок (Workday, Absences, Profiles)

Три связанных раздела:

- **`My Profile`** — расширенный профиль сотрудника (Bitrix-стиль, inline-edit): должность, отдел, телефоны, навыки, дни рождения; редактируется самим пользователем.
- **`Workday`** — рабочее время: старт смены через модальное окно при первом входе в день, чип в верхней панели с накопительным счётчиком за день (МСК-таймзона). Bekend-эндпоинт `/workday/grid` отдаёт посуточно по всем сотрудникам. На странице переключатель «одна строка / табличный вид».
- **`Absences`** — единая таблица отсутствий всех сотрудников + таймлайн. Счётчик пересечений в день показывается только при ≥2 одновременных отсутствий.

Все три гейтятся секцией прав: `profiles`, `workday`, `absences`.

### 6.16 Глобальный поиск и Command Palette

- `Ctrl+K` — командная палитра: быстрый поиск по сделкам/задачам/договорам/сотрудникам/документам/lead'ам/постам ленты/нормам, переход по результатам.
- `/search` — полноценная страница с фильтрацией по entity_type и пагинацией.
- Engine: гибридный (см. INTERNAL §9.2) — FTS5 BM25 + bge-m3 cosine, объединение через RRF, ACL применяется per-row (см. INTERNAL §9.1).

### 6.17 Event Bus v2 (Integrations)

Шина событий для интеграций с внешними системами (Telegram, 1С, Диадок, Калькуляторы, ЭЦП и т.п.):

- transactional outbox (`event_outbox`) — события эмитятся вместе с бизнес-транзакцией;
- подписки (`event_subscription`) с URL, headers, retry-policy и JSON-Logic условием (фильтр на стороне backend'а);
- worker (`event_outbox_worker.py`) — батч-обработка с per-entity ordering, HMAC-подпись, dedup на стороне consumer'а;
- админский UI `Integrations Outbox` / `Subscriptions` для мониторинга и retry;
- каталог событий (~140+ типов) в `docs/events.json` + публичный реф в `docs/EVENTS_API.md`.

### 6.18 Мобильный клиент (Flutter)

Отдельный проект `mobile_app/` — Flutter-клиент для iOS и Android (MVP в `docs/MOBILE_FLUTTER_MVP.md`, iOS-сборка через TestFlight в `docs/IOS_CLOUD_BUILD_TESTFLIGHT.md`). CI через Codemagic (`codemagic.yaml` в корне).

## 7. Интеграции

Система уже интегрирована или подготовлена к интеграциям со следующими внешними сервисами:

- `DaData` — контрагенты, банки и подсказки;
- `Yandex OAuth` — подключение почтовых ящиков;
- `IMAP/SMTP` — синхронизация и отправка писем;
- локальное файловое хранилище;
- Redis для rate limiting и части security-механизмов.

## 8. Безопасность и авторизация

На верхнем уровне сейчас зафиксированы следующие базовые принципы:

- авторизация через JWT;
- токены в `httpOnly` cookies;
- CSRF-защита для cookie-auth сценариев;
- обязательная 2FA для рабочей системы;
- section-based доступ на backend;
- **per-row ACL для child-entity** в гибридном поиске: фильтрация результатов через родительскую сущность, чтобы не утекали данные через child-section с более широкими правами (см. `INTERNAL.md` §9.1);
- дополнительные ограничения для чувствительных write-операций;
- rate limiting для логина и общий API rate limiting;
- security headers и CSP;
- HMAC-подпись исходящих webhook'ов из Event Bus.

Документ `PROJECT_OVERVIEW.md` не заменяет security-аудит, но важно понимать, что система уже не является “простым локальным CRUD-приложением”: security-контур здесь встроен в архитектуру.

## 9. Среды и конфигурация

По умолчанию проект ориентирован на:

- локальную разработку на `SQLite`;
- production-конфигурацию через переменные окружения;
- отдельные frontend build-варианты;
- раздельный запуск backend, frontend и фоновых workers.

Ключевые технические особенности текущей конфигурации:

- `SQLALCHEMY_DATABASE_URI` по умолчанию указывает на локальный `crm.db`;
- `STORAGE_LOCAL_ROOT` определяет файловое хранилище;
- включен `REQUIRE_TWO_FACTOR`;
- frontend собирается через `Vite 6`;
- backend работает на `FastAPI 0.104.x`.

## 10. Что особенно важно помнить о системе

### 10.1 Это не набор несвязанных разделов

Практически все крупные разделы связаны друг с другом:

- товары сделки влияют на этапы, контрактацию и исполнение;
- договоры влияют на контрактацию и оплаты;
- treasury и ДДС влияют на финансовую картину проектов;
- документы и письма связаны с договорами, проектами и заказчиками;
- роли и привязки пользователя влияют на видимость данных в исполнительской и заказческой панелях.

### 10.2 Project Detail — центральная точка системы

Если нужно понять реальную “основу продукта”, смотреть нужно прежде всего на:

- `Projects`
- `ProjectDetail`
- `Contracts`
- `ExecutorPanel`
- `CustomerPanel`
- `Treasury`
- `Outgoing Registry`

Именно через них видно, как сходятся продажи, проектное управление, документы, исполнение и деньги.

### 10.3 Система уже выросла до enterprise-монолита

Текущий проект — это не MVP. Это крупный доменный монолит с:

- десятками роутеров;
- десятками экранов;
- множеством ролей;
- фоновой обработкой;
- внешними интеграциями;
- развитой файловой и документной логикой.

Поэтому любые изменения стоит оценивать не как локальный UI-фикс, а как изменение внутри большой взаимосвязанной бизнес-системы.
